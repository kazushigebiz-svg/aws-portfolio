Direct Connect and On-Prem VM Migration to AWS

Overview

Client: Financial Security Firm

Proposal: Migration of internal systems from on-premises data center to AWS.

  • Establishment of AWS Direct Connect and Site-to-Site VPN connectivity
  • Migration of VMs from Stratus everRun Virtualization Platform to AWS
  • Deployment of resilient architecture on AWS

Existing Arrangement

The customer’s custom-built financial application currently runs on the Stratus everRun Virtualization Platform, which provides fault-tolerant workloads.

  • Challenge: Physical server hardware is reaching end-of-life. The customer must decide between hardware replacement or cloud migration to reduce physical footprint in the colocation data center, while retaining core network devices on-premises.
  • OS Lifecycle: The existing Stratus everRun version runs on CentOS-based Linux, but due to CentOS EOL announced by Red Hat, Stratus everRun is shifting to Ubuntu-based Linux.
  • Infrastructure: Current hardware specifications and infrastructure layout are documented.

Approach

The Stratus everRun platform delivers high availability through fault tolerance. The goal of this migration is to recreate a comparable level of resilience on AWS using native cloud services.

Step 1: VPC and Subnet Setup

  • Create separate VPCs for Production and UAT
  • Configure subnets across multiple Availability Zones

Step 2: Routing Setup

Site-to-Site VPN: Create VGW, configure Customer Gateway (FortiGate), establish IPsec tunnel, routing, connectivity test.

Direct Connect: Provision Private VIF, configure BGP peering, conduct failover and throughput testing.

Note: Direct Connect provisioning requires months of lead time (circuit application, ISP coordination, cross-connect, testing).

Step 3: Instance Setup (EC2 and RDS)

  • Deploy EC2 instances (Windows Server OS) in Production and UAT VPCs
  • Active–Passive failover across separate AZs
  • Three EC2 instances distributed across AZs
  • Amazon RDS for Oracle (Multi-AZ, License Included)

Step 4: Route 53 Setup

Configure Route 53 DNS-level health checks for failover.

Near-zero failover is not required due to low traffic; Route 53 health checks are sufficient.

Step 5: Backup, Monitoring, and Alerts

  • AWS Backup snapshots for EC2 and RDS
  • CloudWatch monitoring and SNS notifications

Features

Reliability & Availability

  • DNS-level failover via Route 53
  • Multi-AZ deployment for EC2 and RDS
  • Automated backups with AWS Backup

Security Enhancements

  • Encrypted connectivity via Direct Connect and Site-to-Site VPN
  • Segregated VPCs for Production and UAT

Cost Optimization

  • Reduced operational overhead by leveraging AWS managed services
  • Right-sized architecture

Business Value

  • Lower hardware footprint and reduced colocation costs
  • Improved scalability and flexibility

Cost & Security Considerations

  • Cost Drivers: Hardware replacement vs AWS migration, licensing, managed services
  • Security Concerns: Cloud migration required careful evaluation by the system department

Hindsight & Lessons Learned

  • Differences between on-prem and AWS architectures, especially connectivity and availability models
  • Connectivity from colocation to AWS via Direct Connect was critical
  • Telecom coordination with ISP was essential for Direct Connect circuits

Extra Note on Terminology

Fault-tolerant architecture: Workloads continue seamlessly even if hardware fails, often requiring specialized infrastructure.

Resilient architecture: Emphasizes recovery and continuity through redundancy, automation, and managed services. It balances availability, cost, and scalability.

This distinction explains why the migration focuses on resilience rather than strict fault tolerance—aligning with cloud-native design principles and cost optimization.
