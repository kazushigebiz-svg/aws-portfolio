Direct Connect and On-Prem VM Migration to AWS

Hybrid connectivity with AWS Direct Connect and Site-to-Site VPN, resilient AWS architecture, and migration from Stratus everRun to AWS.

Hybrid networking AWS Direct Connect Site-to-Site VPN EC2 & RDS Resilience

Overview

Client: Financial Security Firm

Proposal: Migration of internal systems from on-premises data center to AWS.

Scope:

  • Connectivity: Establish AWS Direct Connect and Site-to-Site VPN
  • Workload migration: Move VMs from Stratus everRun Virtualization Platform to AWS
  • Architecture: Deployment of resilient architecture on AWS

Existing arrangement

The customer’s custom-built financial application currently runs on the Stratus everRun Virtualization Platform, which provides fault-tolerant workloads.

  • Challenge: Physical server hardware is reaching end-of-life. The customer must decide between hardware replacement or cloud migration to reduce physical footprint in the colocation data center, while retaining core network devices on-premises.
  • OS lifecycle: The existing Stratus everRun version runs on CentOS-based Linux. Due to CentOS EOL announced by Red Hat, Stratus everRun is shifting to Ubuntu-based Linux.
  • Infrastructure: Current hardware specifications and infrastructure layout are documented.

Approach

The Stratus everRun platform delivers high availability through fault tolerance. The goal of this migration is to recreate a comparable level of resilience on AWS using native cloud services.

Step 1: VPC and subnet setup

  • Environment isolation: Create separate VPCs for Production and UAT
  • Redundancy: Configure subnets across multiple Availability Zones

Step 2: Routing setup

Site-to-Site VPN

  • AWS side: Create a Virtual Private Gateway (VGW) on AWS VPC
  • Customer side: Configure Customer Gateway (FortiGate Firewall at Data Center)
  • Tunnel: Establish IPsec VPN
  • Routing: Configure routes on both ends and validate connectivity

Direct Connect

  • VIF: Provision Private Virtual Interface (VIF) for internal traffic
  • Dynamic routing: Configure BGP peering between Customer Gateway and AWS
  • Validation: Conduct failover and throughput testing

Remarks: Direct Connect and VPN are critical for this hybrid setup. Provisioning can require months (circuit application, ISP coordination, cross-connect, and testing). Plan schedules carefully to avoid implementation delays.

Step 3: Instance setup (EC2 and RDS)

  • Compute: Deploy EC2 instances (Windows Server OS) in Production and UAT VPCs
  • Failover: Configure Active–Passive across separate AZs in Production
  • Distribution: Run three EC2 instances across distinct AZs
  • Database: Deploy Amazon RDS for Oracle (Multi-AZ, License Included) to reduce operational overhead

Step 4: Route 53 setup

  • Health checks: Configure Route 53 DNS-level health checks for failover

Remarks: Near-zero failover is not required due to low internal traffic. Route 53 health checks are sufficient for the failover solution.

Step 5: Backup, monitoring, and alerts

  • Protection: Configure AWS Backup snapshots for EC2 and RDS
  • Observability: Set up CloudWatch monitoring and SNS notifications

Final architecture

The proposed architecture connects the customer’s HQ, branch offices, and data center to AWS via hybrid networking. Additional network infrastructure changes (hardware installation, reconfiguration) are required but not detailed in this summary.

Features

Reliability & availability

  • DNS failover: Route 53 health checks
  • Redundancy: Multi-AZ for EC2 and RDS
  • Backups: Automated snapshots via AWS Backup

Security enhancements

  • Encrypted connectivity: Direct Connect and Site-to-Site VPN
  • Segregation: Separate VPCs for Production and UAT

Cost optimization

  • Managed services: Reduced operational overhead using AWS managed offerings
  • Right sizing: Avoided over-provisioning with a pragmatic resilience model

Business value

  • Lower footprint: Reduced on-prem hardware and colocation costs
  • Scalability: Improved flexibility for future workloads

Cost & security considerations

  • Cost drivers: Hardware replacement vs. AWS migration, licensing, and managed services
  • Security: Cloud migration required careful evaluation by the system department to ensure compliance and data protection

Hindsight & lessons learned

  • Architecture shift: Key differences between on-prem and AWS, especially connectivity and availability models
  • Hybrid focus: Connectivity from the colocation data center to AWS via Direct Connect was a major consideration
  • Telecom coordination: ISP collaboration was essential for Direct Connect circuit delivery and reliable AWS access

Extra note on terminology

Fault-tolerant architecture: Workloads continue seamlessly even if hardware fails, often requiring specialized infrastructure.

Resilient architecture: Emphasizes recovery and continuity through redundancy, automation, and managed services. It may not guarantee zero downtime, but it balances availability, cost, and scalability.

This distinction explains why the migration focuses on resilience rather than strict fault tolerance—aligning with cloud-native design principles and cost optimization.
